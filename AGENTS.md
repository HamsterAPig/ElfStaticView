# ElfStaticView 项目协作规范

## 基本原则

1. 全程使用简体中文沟通、分析、注释和提交说明。
2. 优先执行可验证的实际操作，不用空泛建议替代推进。
3. 遵循 KISS 与 YAGNI：代码保持直接、清晰、可读，不提前实现当前任务不需要的抽象。
4. 任何未经验证的结论都必须明确标注为未验证。
5. 不回滚、不覆盖用户已有改动；遇到无关工作区变更时，只处理当前任务范围。

## 新任务启动流程

每次开始一个全新任务时，必须先执行以下检查：

```powershell
git status --short --branch
git rev-parse --verify HEAD
```

分支规则：

1. 如果仓库已有提交，必须基于当前 `HEAD` 创建任务分支。
2. 分支名按任务类型命名：
   - `task/<short-topic>`：普通任务或功能开发。
   - `fix/<short-topic>`：缺陷修复。
   - `chore/<short-topic>`：工程、构建、文档维护。
3. 如果仓库还没有任何提交，不能声称已经基于当前提交创建分支；应先说明这是初始化仓库状态，并规划初始化提交边界。
4. 创建分支前如发现未提交改动，必须判断是否属于当前任务；不属于当前任务的改动不得擅自纳入提交。

## 任务计划文件

每次任务开始时，必须在系统临时目录写入分步骤计划文件。

Windows 环境使用：

```powershell
$env:TEMP
```

计划文件命名建议：

```text
ElfStaticView-task-plan-<yyyyMMdd-HHmmss>.md
```

计划文件至少包含：

```markdown
# ElfStaticView Task Plan

## Task
<任务名称>

## Branch
<分支名或初始化状态说明>

## Steps
- [ ] 1. 检查仓库状态
- [ ] 2. 创建或确认任务分支
- [ ] 3. 实施修改
- [ ] 4. 运行验证
- [ ] 5. 规划并创建提交

## Verification
- Pending

## Commit
- Pending
```

执行要求：

1. 每完成一个步骤，必须立即回写计划文件，将对应步骤更新为 `[x]`。
2. 任务结束时，必须记录实际运行过的验证命令。
3. 如果创建了提交，必须记录提交 hash。
4. 如果因为仓库状态、测试失败或用户确认缺失而没有提交，必须在计划文件中写明原因。

## 提交规划与自动提交

每次任务开始时，需要预先规划提交粒度；任务完成并验证后，应自动创建符合任务边界的提交。

提交粒度规则：

1. 文档修改：使用一个独立提交，例如 `docs: update agent workflow rules`。
2. 构建系统或目录结构修改：使用独立提交，例如 `build: add cmake project skeleton`。
3. 功能代码修改：按可独立理解的行为边界拆分提交。
4. 测试补充：如果与功能改动紧密相关，可以同提交；如果是大范围测试整理，应单独提交。
5. 不允许把无关文件、历史暂存内容或用户未授权改动混入当前任务提交。

提交前必须执行：

```powershell
git diff --check
git status --short --branch
```

如果本次任务涉及 CMake 工程或 C++ 代码，至少执行：

```powershell
cmake -S . -B build -G "Ninja"
cmake --build build
ctest --test-dir build --output-on-failure
```

后台测试命令建议设置 60 秒超时，除非当前任务明确需要更长时间。

## 构建系统与工具链

1. 使用 Modern CMake 管理工程。
2. 默认使用 GCC 工具链。
3. C++ 标准使用 C++2a。
4. CMake 配置优先使用 target 级写法，避免全局污染。
5. 不在源码中硬编码本机绝对路径。
6. 第三方依赖必须有清晰边界，不把依赖细节扩散到业务模块。

CMake 编写要求：

1. 优先使用 `target_compile_features`、`target_compile_options`、`target_include_directories`、`target_link_libraries`。
2. 可执行程序、库、测试目标职责清晰。
3. 公共编译选项集中管理，但不要过早抽象复杂 CMake 框架。
4. 新增 CMake 文件时，应保证从干净构建目录可以重新配置和构建。

## 推荐目录结构

初始源码目录按真实领域职责划分，保持克制：

```text
.
├── CMakeLists.txt
├── cmake/
│   ├── toolchains/
│   └── modules/
├── include/
│   └── elf_static_view/
├── src/
│   ├── elf/
│   ├── analysis/
│   ├── config/
│   ├── logging/
│   └── main.cc
├── tests/
│   ├── elf/
│   ├── analysis/
│   └── fixtures/
├── 3rdparty/
└── docs/
```

目录职责：

1. `src/elf`：ELF / DWARF 读取与解析封装。
2. `src/analysis`：静态分析、索引、结构化视图数据生成。
3. `src/config`：配置模型与 YAML 读取。
4. `src/logging`：spdlog 初始化与日志入口。
5. `src/main.cc`：程序入口，只做参数解析、初始化和启动编排。
6. `include/elf_static_view`：稳定对外接口；内部实现细节不要提前暴露。
7. `tests`：单元测试、集成测试和 fixture 数据。
8. `cmake`：工具链、编译选项和辅助 CMake 模块。

禁止事项：

1. 不提前创建职责不清的 `src/core`。
2. 只有在实际出现 GUI 或 TUI 需求时，才创建 `src/ui`。
3. 只有在启动流程复杂到需要独立编排时，才创建 `src/app`。

## 代码质量要求

1. 代码优先可读性和可维护性，不做过度封装。
2. 核心流程、关键算法、重要边界条件必须写简体中文注释。
3. 命名应表达真实领域含义，避免无意义缩写。
4. 新增核心逻辑必须考虑可测试性。
5. 模块之间依赖方向应清晰：分析层可以消费 ELF 解析结果，ELF 解析层不反向依赖分析层。
6. 公共接口要稳定、简洁；内部实现可以按需要迭代，不提前暴露。

## 测试与验证

1. 新增核心逻辑时，应同步补充单元测试。
2. 修复缺陷时，应优先补充能复现缺陷的测试。
3. 测试数据放在 `tests/fixtures`，避免散落到源码目录。
4. 后台执行测试时建议设置最大超时时间 60 秒，避免任务卡死。
5. 若验证无法执行，必须说明具体命令、失败原因和剩余风险。

## 危险操作确认

执行以下操作前必须获得用户明确确认：

1. 删除文件或目录。
2. 批量移动或批量重命名文件。
3. 修改系统环境变量、权限或全局配置。
4. 全局安装、卸载或升级包管理依赖。
5. 删除数据库、结构变更或批量数据更新。
6. 调用生产环境 API 或发送敏感数据。

确认格式：

```text
危险操作检测！

操作类型：<具体操作>
影响范围：<详细说明>
风险评估：<潜在后果>

请确认是否继续？需要明确回复“是”、“确认”或“继续”。
```
